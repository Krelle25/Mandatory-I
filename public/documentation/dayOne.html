JavaScript Fundamentals

Variables: const vs let

A key rule introduced on day one:

  • Use const by default
  • Use let only when reassignment is required

const luckyNumber = 4;
console.log(luckyNumber);

Using const prevents reassignment of the variable itself. However, when working with objects and arrays, their contents can still be changed.

Objects

Objects store data as key-value pairs.


const person = {
    name: "Kristian"
};

person.name = "Knud";
person.age = 82;

This demonstrates that:

  • Object properties can be changed
  • New properties can be added
  • Declaring an object with const does not make it immutable

Arrays

Arrays are also mutable, even when declared with const


const people = [];
people.push("Krelle");

Although the array itself cannot be reassigned, its contents can still be changed and modified

JavaScript Data Types

The following JavaScript data types were introduced:

  • String = textual value
  • Number = numeric value
  • Boolean = logical values (true or false)
  • BigInt = very large integers
  • Object = collection of key-value pairs
  • Symbol = unique and immutable value
  • null = intentional absence of value
  • undefined = declared variable but yet to be assigned a value

JSON vs JavaScript Objects

The difference between JSON and JavaScript objects were introduced and explained

JSON requires quotation marks ("") around the key


{
    "name": "Kristian"
}

JavaScript objects do NOT require quotation marks around the key


const person = {
    name: "Kristian"
}

Type Coercion & Equality

Type Coercion

JavaScript can automatically convert values from one type to another when performing operations or comparisons. This behavior is called type coercion. While sometimes convenient, implicit type coercion can lead to unexpected results and subtle bugs. 


const year = "2025";
const increment = 1;
const currentYearFirst = year + increment;
// result = 20251, and this is because of "implicit type coercion"

To avoid this:


const currentYearFirst = Number(year) + increment;
const currentYearSecond = parseInt(year) + increment;
const currentYearThird = +year + increment;
// result = 2026 because of converting the string to a number

Equality

JavaScript provides two ways to compare values for equality: == (loose equality) and === (strict equality). The difference is how JavaScript handles type conversion during the comparison. 


5 == "5"   // true
5 === "5"  // false

Loose equality (==) performs implicit type coercion before comparing the values. Strict equality (===) does not perform type conversion and requires both the value and the type to match.

  • Prefer strict equality ===, which compares both value and type
  • Avoid loose equality ==, as it performs implicit type conversion
  • Convert values when needed (parseInt(), parseFloat())

String in JavaScript

There are three common ways to define strings in JavaScript:

  • "double quotes"
  • 'single quotes'
  • `template literals`

Template literals allow string interpolation, which makes it easier to embed variables inside strings:


const firstName = "Kristian";
const lastName = "Steffensen";

`My first name is ${firstName} and my last name is ${lastName}`

Automatic Semicolon Insertion (ASI)

JavaScript automatically inserts semicolons in many cases. This feature is called "Automatic Semicolon Insertion". Although this is a smart feature, relying on ASI is discouraged, as it can sometimes lead to unexpected behavior. Writing clear and consistent code is preferred.

REST API Design Exercise

Topic: Beers

The task was to design a CRUD based REST API on the topic 'Beers'.

Design Goals:

  • Follow core REST conventions
  • Use the correct HTTP method for each action
  • Use clear, pluralized resource names for endpoints
  • Ensure endpoints represent a collection of resources

HTTP Methods

Each CRUD operation maps to a specific HTTP method. Using the correct method helps make APIs predictable, consistent, and easier to understand.

  • GET → Retrieve data from the server
  • POST → Create a new resource
  • PUT → Replace an existing resource
  • PATCH → Partially update an existing resource
  • DELETE → Remove a resource

CRUD Operations

REST APIs usually follow the CRUD model, which describes the four basic things you typically want to do with data in an application.

  • Create → POST
  • Read → GET
  • Update → PUT/PATCH
  • Delete → DELETE

When documenting or implementing REST APIs, it is helpful to keep endpoints in a consistent order. Typically, endpoints are grouped by resource and listed in the same logical order as the CRUD operations: GET, POST, PUT/PATCH, and DELETE.

Example: API Design (Beer)

In REST APIs, endpoints represent resources (nouns) rather than actions (verbs). For example, /beers represents the beer collection, while /beers/{id} represents a specific beer.

Method Endpoint Description
GET/beersRetrieve all beers
GET/beers/{id}Retrieve a beer by id
POST/beersCreate a new beer
PUT/beers/{id}Update an entire beer
PATCH/beers/{id}Update parts of a beer
DELETE/beers/{id}Delete a beer
